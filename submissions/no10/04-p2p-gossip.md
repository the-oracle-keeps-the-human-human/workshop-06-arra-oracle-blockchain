# บทที่ 4: Peer-to-Peer (P2P) Gossip & Sync Paths

เส้นทางการเชื่อมต่อระหว่างโหนดผ่าน P2P Gossip เพื่อดึง unsafe head ในภาวะที่ยังไม่มี L1 batcher

---

## 4.1 Sync Paths สองแบบของ OP Stack: P2P Gossip vs. L1 Derivation

สำหรับสถาปัตยกรรมการประสานข้อมูลของ OP Stack นั้น
เส้นทางการ sync จะถูกแบ่งออกเป็นสองรูปแบบหลักอย่างชัดเจน
เมื่อเราศึกษาโครงสร้างการทำงาน
ก็จะเข้าใจการแยกส่วนกันระหว่างสองเลเยอร์นี้นะครับ

- **P2P Gossip (Unsafe Path)**: เส้นทางข้อมูลความเร็วสูง
  เมื่อ sequencer บังเกิดบล็อก L2 ล่าสุดขึ้นมา
  ก็จะเผยแพร่ข้อมูลอย่างรวดเร็วผ่านเครือข่าย libp2p
  ส่งตรงไปยังโหนด follower ทันที
  บล็อกเหล่านี้จะมีสถานะเป็น unsafe head
  ทำให้ปลายทางสามารถรับรู้การเคลื่อนไหวได้ทันตาเห็นละนะ

- **L1 Derivation (Safe Path)**: เส้นทางฉันทามติหลัก
  เมื่อ `op-batcher` รวบรวมข้อมูล L2 transaction batches 
  แล้วบันทึกลงไปยัง L1 Sepolia chain สำเร็จ
  โหนด `op-node` ก็จะดึงข้อมูลดังกล่าวมาสร้างบล็อกใหม่
  ทำให้บล็อกเปลี่ยนสถานะเป็น safe head
  และก้าวสู่ความเป็น finalized block ในที่สุดนั่นเอง

แต่ทว่า ในการซ้อมรันช่วงเริ่มต้นของพวกเรานั้น
เมื่อระบบยังไม่มี `op-batcher` คอยส่งข้อมูลขึ้น L1
ผลลัพธ์เป็นอย่างไรกันนะ?
ก็บล็อกในฝั่ง `safe_l2` จะหยุดนิ่งอยู่ที่เลข 0 ซะงั้น
ขณะที่บล็อก `unsafe_l2` ของ Nova วิ่งล่วงหน้าไปไกลแล้ว
สถานการณ์เช่นนี้ หากเผลอปิดการใช้งาน P2P ไว้
ตัวโหนด follower ก็จะค้างเติ่งอยู่ที่บล็อก 0 ทันทีเลยสิ

พอโหนดค้างแบบนี้ หลายคนก็มักจะเข้าใจผิด
คิดว่าการแก้ไขต้องไปยุ่งกับ geth devp2p ของ `op-geth`
แต่ความเป็นจริงนั้นหาใช่ไม่
เพราะใน OP Stack ตัวบล็อก L2 จะถูกกำหนดโดย `op-node`
แล้วจึงส่งผ่านคำสั่งเข้า Engine API ของ `op-geth` เท่านั้น
ดังนั้น หากต้องการให้บล็อก L2 ขยับ
ก็ต้องหันมาจัดการที่ระบบ P2P ของ `op-node` เถอะนะ

## 4.2 การใช้ libp2p multiaddr แทน enode (geth devp2p)

สำหรับระบบจับคู่ Peer ของโหนดใน Ethereum ทั่วไป
เรามักคุ้นเคยกับรูปแบบ `enode` ของ geth devp2p
แต่เมื่อต้องตั้งค่าโหนดบน OP Stack Rollup
การเชื่อมต่อระหว่าง `op-node` จะเปลี่ยนมาใช้ libp2p แทนนะครับ

ความสับสนตรงนี้เคยนำพาความปวดหัวมาสู่พวกเรา
เมื่อ Vessel และ Weizen พยายามเชื่อมต่อเข้ากับ reference chain
แต่โหนดกลับนิ่งสนิทไม่ตอบสนองใดๆ
พอ Leica และ Orz เข้ามาช่วยตรวจสอบตอน 08:35 UTC 
ก็ชี้ทางสว่างให้พวกเราได้เข้าใจ
ว่า libp2p multiaddr นั้นไม่ใช่ enode ซะงั้น

เนื่องจาก `op-node` ถูกเขียนขึ้นด้วย go-libp2p
ส่วน geth นั้นรันด้วยระบบ devp2p ดั้งเดิม
เมื่อเรานำ enode URL ไปยัดใส่ช่องค้นหา static peer ของ `op-node`
ตัว parser ก็จะปฏิเสธและทิ้งการเชื่อมต่อนั้นทันทีเลยนะ

ดังนั้น การกำหนด peer จึงต้องใช้รูปแบบของ multiaddr เสมอ
ซึ่งจะมีหน้าตาเฉพาะตัวดังนี้ครับ:

```text
/ip4/<IP_ADDRESS>/tcp/<PORT>/p2p/<NODE_PUBLIC_KEY>
```

ตัวอย่างเช่น:
```text
/ip4/141.11.156.4/tcp/9222/p2p/16Uiu2HAm...
```

เมื่อใช้รูปแบบที่ถูกต้องนี้แล้ว
ตัวโหนด `op-node` จึงจะสามารถค้นหาและสร้างท่อเชื่อมต่อ
เพื่อดึงบล็อก unsafe จาก Nova Reference Chain ได้สำเร็จละนะ

## 4.3 ข้อควรระวัง: flag `--p2p.disable` และ port collision บนโฮสต์เดียวกัน

ในห้องเรียนวันนั้นที่สมาชิกร่วมกันแก้ปัญหา
เมื่อมีใครสักคนเจอปัญหา โหนดอื่นก็ส่งเสียงสะท้อนตามกันมา
พอสืบเสาะจึงพบข้อควรระวังสำคัญสองประการครับ

ประการแรกคือการเผลอใช้งาน flag `--p2p.disable`
เมื่อใส่ flag นี้ลงไปในคำสั่งรันโหนด
ระบบจะปิดการรับส่งข้อมูลผ่าน Gossip ทันที
หากโครงการยังไม่มี batcher ส่งข้อมูลขึ้น L1
โหนดของเราก็จะไม่ได้รับบล็อกใดๆ เลย
โหนดจึงติดแหง็กอยู่ที่บล็อกเริ่มต้นไม่ยอมขยับไปไหน
การเปิดใช้ P2P และกำหนด static peer ให้ถูกต้อง
จึงเป็นกุญแจสำคัญสำหรับระยะแรกเริ่มนี้เถอะนะ

ประการต่อมาคือเรื่อง Port Collision บนเครื่องเดียวกัน
เมื่อผู้เรียนต้องการรันทั้ง sequencer และ follower
ไว้บนเครื่องเซิร์ฟเวอร์เดียวกันในเวลาพร้อมๆ กัน
หากไม่วางแผนจัดพอร์ตให้ดี พอร์ต P2P (เริ่มต้น 9222) ก็จะชนกัน
ทำให้โหนดหลังเปิดพอร์ตรับ Gossip ไม่ขึ้นซะงั้น

ดังนั้น เพื่อหลีกเลี่ยงเหตุการณ์ข้างต้น
ผู้จัดทำจึงได้สรุปสูตรขั้นต่ำสำหรับรัน follower ดังนี้ครับ:

1. นำไฟล์ `genesis.json` และ `rollup.json` ของ Nova มาใช้
2. ตั้งค่าไฟล์ `jwt.txt` เพื่อเชื่อมโยง Engine API ของตนเอง
3. ห้ามระบุ `--sequencer.enabled` ใน follower node
4. ตรวจสอบให้มั่นใจว่าไม่มี flag `--p2p.disable`
5. ระบุ `--p2p.static` ด้วย libp2p multiaddr ของ Nova
6. วางระบบพอร์ตไม่ให้ซ้ำซ้อนกับโหนดอื่นบนเครื่องเดียวกัน

เมื่อโหนดของเพื่อนอย่าง Vessel และ Weizen ได้รับการปรับแต่ง
เปลี่ยนรูปแบบ Peer และเปิดพอร์ต Gossip ได้อย่างถูกต้อง
ข้อมูล unsafe head จาก Nova ก็หลั่งไหลเข้ามา
ทำให้ L2 ขยับก้าวพ้นบล็อก 0 ได้สำเร็จในที่สุดนั่นเองนะครับ
