# บทที่ 6: Checklist & Best Practices for New Rollups

สรุปขั้นตอนและแนวทางปฏิบัติตามลำดับขั้น (The Algorithm)
สำหรับผู้ที่ต้องการสร้างเชนใหม่
เพื่อป้องกันความสับสนและข้อผิดพลาดในการรันจริง

---

## 6.1 ลำดับขั้นการลบส่วนเกินก่อนเร่งความเร็ว (Delete before Automate)

หลักการพัฒนา 5 ขั้น (The Algorithm) ของ Master Bo
กำหนดขั้นตอนไว้อย่างชัดเจน
เมื่อจะเริ่มสร้างระบบ L2 Rollup ใหม่
ก็ต้องทำตามลำดับอย่างเคร่งครัดนะครับ

ขั้นตอนแรก ปรับความต้องการให้ไม่โง่ (Make requirements less dumb)
ความต้องการทุกอย่าง ต้องมีชื่อคนรับผิดชอบชัดเจน
หากไม่รู้ว่าใครเป็นคนสั่ง ก็ต้องตัดทิ้งไปเสียก่อน
เมื่อขอบเขตชัดเจน จึงค่อยเริ่มทำงานจริง

ขั้นตอนที่สอง ลบขั้นตอนหรือชิ้นส่วนส่วนเกิน (Delete the step / part)
สิ่งนี้สำคัญที่สุดในกระบวนการทั้งหมด
หากลบออกได้ ก็ต้องลบให้มากที่สุดเท่าที่จะทำได้
เมื่อรู้สึกว่าลบมากเกินไปจนต้องเพิ่มกลับมาสักสิบเปอร์เซ็นต์
นั่นแหละจึงจะพอดี
แต่คนทั่วไป มักจะข้ามขั้นตอนนี้ไปอย่างน่าเสียดาย

ขั้นตอนที่สาม ปรับปรุงส่วนที่เหลือให้เรียบง่าย (Simplify & Optimize)
เมื่อตัดส่วนเกินออกไปหมดแล้ว ก็เริ่มปรับแต่งขั้นตอนที่เหลือ
ทำสิ่งที่มีอยู่ให้ทำงานได้ดีและประหยัดทรัพยากร
แต่ห้ามปรับแต่งสิ่งที่ไม่ควรมีตั้งแต่แรกเด็ดขาดนะครับ

ขั้นตอนที่สี่ เร่งความเร็วการทำงาน (Accelerate)
เมื่อระบบเรียบง่าย ก็เร่งรอบการผลิตบล็อกและการประมวลผล
ทำรอบการทำงานให้เร็วขึ้นถึงขีดสุด

ขั้นตอนสุดท้าย ทำให้อัตโนมัติ (Automate)
หากข้ามไปทำขั้นตอนอัตโนมัติก่อนลบส่วนเกิน
ระบบอัตโนมัติก็จะมีแต่ความซับซ้อนที่ไร้ประโยชน์นั่นเอง

บทเรียนเรื่องระบบตรวจสอบพฤติกรรม (Behavior vs Vocabulary)
เมื่อศึกษากรณีศึกษาจาก Orz เรื่องความผิดพลาดซ้ำซาก (Honest Failure)
ความเข้าใจในบทเรียนแบบข้อเขียน (Prose)
ก็เป็นเพียงคำศัพท์ประดับสมอง (Vocabulary) เท่านั้น
แต่บทเรียนที่แท้จริง ต้องเปลี่ยนเป็นพฤติกรรมของระบบ (Behavior)
ด้วยการแก้ไขไฟล์ตั้งค่าหรือเขียนสคริปต์ควบคุมการทำงาน

พอเขียนคู่มือเตือนใจไว้ใน markdown
ก็มักจะลืมอ่านเมื่อเวลาผ่านไป
แต่พอเปลี่ยนมาแก้ไฟล์ `settings.local.json` ให้บังคับใช้
หรือเขียนสคริปต์ตรวจสอบก่อนทำ pre-commit (hook script)
ระบบก็ถูกบังคับให้ปฏิบัติตามโดยอัตโนมัติ
นี่แหละคือการติดตั้งบทเรียนลงในพฤติกรรมอย่างแท้จริงนะครับ

## 6.2 Checklist ขั้นต่ำในการเตรียมโหนดและคีย์สำหรับการรันจริง

เมื่อจะเริ่มต้นรันโหนดใหม่ของระบบ L2 Rollup
ผู้ดูแลระบบ ก็ต้องเตรียมการตามรายการตรวจสอบขั้นต่ำนี้ครับ

### รายการตรวจสอบความพร้อมก่อนติดตั้งโหนด (Pre-flight Checklist)

1. การตั้งค่า L1 Contracts และคอนฟิกที่ถูกต้อง
   - ไฟล์ `rollup.json` และ `genesis.json` ต้องมีค่าตรงกับโหนด sequencer
   - หากใช้ไฟล์ต่างชุดกัน โหนดผู้ตาม (follower) ก็จะไม่สามารถซิงก์ได้

2. คีย์และรหัสผ่านสำหรับการสื่อสารภายใน (Engine API Key)
   - ไฟล์ `jwt.txt` สำหรับยืนยันตนระหว่างโหนด
   - โหนด `op-geth` และ `op-node` ก็ต้องใช้ไฟล์คีย์ร่วมกันเพื่อความปลอดภัย

3. การเตรียมรัน Execution Client (op-geth)
   - เปิดใช้งานพอร์ตสำหรับ Engine API (พอร์ต 8551)
   - ป้อนพารามิเตอร์ `--authrpc.jwtsecret` ชี้ไปยังไฟล์คีย์ร่วมกัน
   - ปิดความสามารถในการค้นหาโหนดอัตโนมัติหากใช้ในระดับจำกัด

4. การเตรียมรัน Consensus Client (op-node)
   - ปิดโหมดผลิตบล็อกเพื่อรันเป็นผู้ตาม (`--sequencer.enabled=false`)
   - กำหนดปลายทาง static peer โดยระบุผ่าน `--p2p.static`
   - เมื่อต้องการซิงก์อย่างรวดเร็ว ก็ต้องเปิดใช้งาน P2P Gossip

---

### ทำไมโหนดผู้ตามถึงค้างที่บล็อก 0?

เมื่อเปิดระบบแล้วพบว่าบล็อกของโหนดผู้ตามค้างอยู่ที่บล็อก 0
พอตรวจสอบอย่างละเอียด ก็มักเกิดจากสาเหตุดังนี้ครับ

- การใช้ที่อยู่โหนดผิดรูปแบบ:
  การซิงก์บล็อกของ `op-node` จะเชื่อมต่อผ่านโปรโตคอล libp2p
  หากเผลอระบุที่อยู่ peer เป็นรูปแบบ `enode://` ของ geth
  โหนด `op-node` ก็จะไม่สามารถเชื่อมต่อกันได้ครับ
  ที่อยู่ที่ถูกต้อง ต้องอยู่ในรูปแบบ multiaddr เช่น:

```text
/ip4/10.0.0.1/tcp/9003/p2p/16Uiu2HAm...
```

- ความไม่เข้ากันของข้อมูลตั้งต้น:
  หากใช้ `genesis.json` แตกต่างไปแม้แต่บิตเดียว
  ระบบการคำนวณ hash ก็จะปฏิเสธ payload ทันที

- การปิดใช้งาน P2P Gossip:
  หากเผลอใส่ flag `--p2p.disable` ในตอนสตาร์ตโหนด
  โหนดก็จะไม่สามารถรับบล็อกใหม่ล่าสุด (unsafe block) ได้เลย
  แต่หากเปิด P2P ไว้ ก็จะได้รับบล็อกทันทีที่มีการผลิตเกิดขึ้น

- การขาดข้อมูลบน L1 Sepolia:
  เมื่อไม่มี `op-batcher` คอยส่งธุรกรรม L2 ขึ้นไปยัง L1
  การDeriveบล็อกแบบ Canonical ก็จะไม่สามารถทำได้
  ทำให้ค่าบล็อกปลอดภัย (safe_l2) ค้างอยู่ที่ 0 ตลอดเวลาครับ

## 6.3 Social Coordination: ความร่วมมือระหว่างทีมคือสิ่งขับเคลื่อนเครือข่าย

การสร้างเครือข่ายบล็อกเชนใหม่ ไม่ใช่เรื่องของคนคนเดียว
เมื่อเกิดปัญหาระบบทำงานไม่สอดคล้องกัน
ความล่มสลายของระบบเดี่ยว ก็จะผลักดันให้ทุกคนหันมาปรับปรุงร่วมกันครับ

ชัยชนะที่แท้จริงของทีมงาน Oracle Council ในงานนี้
ไม่ใช่การที่ใครคนใดคนหนึ่งสามารถรันโหนดขึ้นได้เพียงลำพัง
แต่คือการประสานงานของทีม เพื่อเปลี่ยนความเข้าใจ (mental model)
ให้ถูกต้องตรงกันทั้งทีมงานนั่นเอง

เมื่อเปลี่ยนความคิดร่วมกันได้แล้ว ปัญหาต่างๆ ก็จะเริ่มคลี่คลาย

- การแยกประเภทเครือข่าย:
  เข้าใจตรงกันว่า L1 Dev Chain ไม่ใช่ OP Stack L2
  เมื่อแยกบทบาทระหว่าง execution client และ consensus client ออกจากกัน
  โครงสร้างการสื่อสารผ่าน Engine API จึงจะสมบูรณ์

- การปรับเปลี่ยนช่องทางเชื่อมต่อ:
  ยอมรับร่วมกันว่า `op-node` ใช้ libp2p ในการสื่อสาร
  เมื่อเลิกใช้ enode แบบเก่า หันมาจับคู่โหนดด้วย multiaddr
  บล็อกที่ติดขัด ก็เริ่มไหลไปสู่ผู้ตามได้ทันที

- การขับเคลื่อนการเงินของเครือข่าย:
  เมื่อ L2 เริ่มต้นด้วยสถานะว่างเปล่า (Empty Economy)
  ทีมงานก็ช่วยกันบริดจ์เหรียญจาก L1 ผ่าน OptimismPortal
  ทำให้ระบบมีค่าแก๊สหมุนเวียนได้ในที่สุด

บทบาทของเหล่า Fleet Oracles ในความร่วมมือครั้งนี้:

- Nova: จัดการดีพลอย L1 Contracts และดูแลโหนดหลัก (sequencer)
- Orz และ Leica: ศึกษาข้อมูลพอร์ตเชื่อมต่อ และนำเสนอระบบ multiaddr
- Vessel, Weizen และ Atom: ทดสอบระบบผู้ตามและช่วยกันหาจุดบกพร่อง
- SomBo, No.10 X, Tonk และ Mac.1: บริหารเงินทุนค่าแก๊สและบริดจ์เหรียญ

ความร่วมมือทั้งหมดนี้เป็นกลไกตรวจสอบทางสังคม (Social Coordination)
เมื่อเพื่อนร่วมงานช่วยกันทักท้วงและแนะนำจุดที่ถูกต้อง
ก็จะเป็นการควบคุมพฤติกรรมการทำงานร่วมกันผ่านสายตาของผู้อื่น
ช่วยให้ข้อผิดพลาดได้รับการแก้ไขอย่างทันท่วงทีครับ
