# บทที่ 1: The L2 Architecture: Geth Clique vs. OP Stack Rollup

บทเรียนการก้าวข้ามจาก Standalone Clique Consensus สู่ระบบ L2 Rollup ที่แท้จริง

---

## 1.1 Clique PoA vs. OP Stack: สถาปัตยกรรมต่างชั้น

เมื่อเริ่มต้นศึกษาการสร้างบล็อกเชนแบบ L2 Rollup
ความคุ้นเคยแรกสุดของนักพัฒนา
มักจะจบลงที่ระบบ geth Clique PoA
ซึ่งเป็นระบบที่ใช้งานง่ายและรันเดี่ยวได้ทันทีครับ

ในการทดลองของเวิร์กชอป 06 เมื่อวันที่ 19 มิถุนายน 2026
หลายทีมพัฒนา เช่น ทีม 2, 4, 5, 6, 7, 8 และ 11
ก็เลือกที่จะติดตั้งระบบ geth Clique PoA
เนื่องจากสามารถตั้งค่าและให้ผลลัพธ์ได้อย่างรวดเร็ว
แต่เมื่อนำส่งงานไปให้ผู้คุมระบบตรวจสอบ
ก็ต้องหยุดชะงักกับคำถามสำคัญจากพี่นัท (P'Nat)
ซึ่งได้ทักท้วงทฤษฎีการติดตั้งเมื่อเวลา 07:26 UTC ว่า:

> "เราไม่ต้องใช้ OP node หรอวะ มันต้องใช้ดิ"

คำทักท้วงนี้  ทำให้ทุกทีมเริ่มฉุกคิด
และหันมามองความแตกต่างทางสถาปัตยกรรมอย่างจริงจังครับ
เมื่อไม่มี op-node มาทำหน้าที่เชนคอนเซนซัสร่วมกับ L1
บล็อกเชนที่รันอยู่ก็เป็นได้เพียง Standalone Chain ธรรมดา
ไม่ได้มีความเกี่ยวข้องใดๆ กับความปลอดภัยของ L1 เลยนะ

แต่ถ้าเปลี่ยนมาใช้ระบบ OP Stack Rollup
รูปแบบการทำงานจะต่างออกไปอย่างสิ้นเชิงครับ
เนื่องจาก OP Stack แยกส่วนการประมวลผลธุรกรรม (Execution)
ออกจากส่วนการตกลงพ้องใจของระบบ (Consensus)
โดยนำความปลอดภัยและกลไกรับรองความถูกต้อง
ไปยึดโยงอยู่กับข้อมูลบนเชนหลัก (L1) เสมอครับ

ความแตกต่างในระดับรากฐานนี้
ทำให้อธิบายหลักการทำงานของทั้งสองระบบได้ดังนี้:

### geth Clique PoA (Proof of Authority)
เมื่อกำหนดตัว Signer หรือบัญชีที่มีสิทธิ์เซ็นบล็อกไว้แล้ว
ตัวโปรแกรม geth ก็จะทำหน้าที่ผลิตบล็อกตามเวลาที่กำหนด
โดยใช้กุญแจส่วนตัวของ Signer ลงนามรับรองโดยตรง
แต่จุดอ่อนสำคัญคือระบบนี้รันเป็นเอกเทศ
ไม่มีกลไกยืนยันหรือตรวจสอบจากเครือข่ายภายนอกเลยครับ
พอโหนดล่มหรือคีย์หลุด  เครือข่ายก็ล้มเหลวทันทีนะ

### OP Stack Rollup
เมื่อข้อมูลธุรกรรมบน L2 ถูกสร้างขึ้น
ระบบจะทำการส่งข้อมูลไปเก็บบน L1 ในรูปของ Batch
พอข้อมูลไปบันทึกบน L1 อย่างถูกต้องแล้ว
ใครก็สามารถมาอ่านและถอดรหัสเพื่อสร้างสถานะ L2 คืนมาได้
กลไกนี้ทำให้ L2 มีความปลอดภัยเทียบเท่ากับ L1 ทันทีครับ
เมื่อผู้ใช้ต้องการย้ายทรัพย์สิน
สัญญาสมาร์ทคอนแทร็กต์บน L1 (เช่น OptimismPortal)
ก็จะเป็นผู้รับประกันการทำธุรกรรมบริดจ์นี้ด้วยครับ

ตารางเปรียบเทียบความแตกต่างด้านสถาปัตยกรรม:

| คุณสมบัติ | Geth Clique PoA | OP Stack Rollup |
| :--- | :--- | :--- |
| **จำนวนโปรแกรมที่รัน** | geth ตัวเดียว | op-node + op-geth |
| **แหล่งที่มาของคอนเซนซัส** | Signer ใน Genesis | Layer 1 (Sepolia) |
| **การเชื่อมต่อกับ L1** | ไม่มี (Standalone) | เชื่อมผ่าน L1 Contracts |
| **การซิงก์ข้อมูลหัวบล็อก** | ผ่าน P2P ของ geth (devp2p) | ผ่าน Engine API และ P2P |
| **การฝาก-ถอนข้ามเชน** | ต้องเขียน Bridge เองภายนอก | มี OptimismPortal ในตัว |

พอพิจารณาตารางเปรียบเทียบนี้แล้ว
ก็เข้าใจได้ทันทีว่าการรัน L2 ที่สมบูรณ์
ไม่ใช่แค่การเปิดเชนเดี่ยวแล้วเปลี่ยนชื่อพารามิเตอร์ครับ
แต่เป็นการสร้างความไว้วางใจข้ามเชนอย่างเป็นระบบครับ

## 1.2 Execution Client (op-geth) และ Consensus Client (op-node)

เมื่อก้าวเข้าสู่ระบบ OP Stack
การรันโหนดจึงต้องทำเป็นสองส่วนประกอบกันครับ
โดยแบ่งการทำงานออกเป็นสองระดับและสองโปรแกรมดังนี้:

### op-geth (Execution Client)
เรื่องการรันธุรกรรมและการจัดการ EVM
เป็นบทบาทหลักของโปรแกรม op-geth ครับ
โปรแกรมตัวนี้ดัดแปลงมาจาก go-ethereum
โดยตัดฟังก์ชันการสร้างบล็อกแบบเดิมออกไป
เพื่อรอรับชุดคำสั่งประมวลผลบล็อกผ่านพอร์ตเฉพาะ
เมื่อผู้ใช้งานส่งธุรกรรมเข้ามาในระบบ
op-geth ก็จะรวบรวมไว้ใน Mempool รอคอยคำสั่ง
และให้บริการ RPC JSON API มาตรฐาน (เช่น eth_*) ครับ

### op-node (Consensus Client)
แต่ผู้ที่ทำหน้าที่ตัดสินใจและดึงข้อมูลคอนเซนซัสคือ op-node
ทำหน้าที่เปรียบเสมือน Beacon Node ของเครือข่าย L2
โดยจะเข้าหา L1 RPC เพื่อคอยอ่านธุรกรรม deposit
จากนั้นนำมารวบรวมเพื่ออนุมานเป็นหัวบล็อกใหม่ (Derivation)
ในกรณีที่เป็น Sequencer
op-node ก็จะเป็นผู้เสนอหัวบล็อก (Unsafe Block) ทุกๆ 2 วินาที
พร้อมส่งต่อไปยัง op-geth เพื่อประมวลผลจริงครับ
รวมถึงให้บริการ API เฉพาะ เช่น `optimism_syncStatus` ด้วยนะ

แผนภาพจำลองสถาปัตยกรรมการสื่อสารระหว่างโหนด:

```
          L1 Sepolia Testnet (RPC / Beacon API)
                           │
                           │ 1. อ่านธุรกรรมและหัวบล็อก L1
                           ▼
     ┌───────────────────────────────────────────┐
     │        op-node (Consensus Client)         │
     │  RPC Port: 8655 | P2P Port: 9222 (libp2p) │
     └─────────────────────┬─────────────────────┘
                           │
                           │ 2. Engine API (Port: 8664)
                           │    ยืนยันตัวตนด้วย JWT
                           ▼
     ┌───────────────────────────────────────────┐
     │        op-geth (Execution Client)         │
     │  RPC Port: 8555 | P2P Port: 30315 (devp2p)│
     └───────────────────────────────────────────┘
```

### หลุมพรางสุดคลาสสิก: ปัญหา Diagnostic Blindness
เมื่อลองใช้งานในเวิร์กชอป 06
ทีม No.10 X และ Atom ก็พบอุปสรรคสำคัญในการตรวจสถานะครับ
พออยากทราบว่าระบบซิงก์ไปถึงบล็อกที่เท่าไรแล้ว
ก็ทดลองยิงคำสั่ง curl ไปที่ RPC ของ op-geth (พอร์ต 8555)
โดยระบุ method เป็น `optimism_syncStatus`
แต่ผลลัพธ์ที่ได้กลับกลายเป็นข้อผิดพลาดดังนี้:

```json
{
  "error": {
    "code": -32601,
    "message": "the method optimism_syncStatus does not exist"
  }
}
```

ความผิดพลาดนี้เกิดจากการเข้าใจสับสนระหว่างบทบาทพอร์ตครับ
เนื่องจาก `optimism_syncStatus` เป็น RPC ของฝั่ง op-node
แต่ทีมงานกลับไปยิงใส่ op-geth ซึ่งไม่มี method นี้รองรับ
พอเรียกพอร์ตใหม่ให้ถูกต้องไปยังพอร์ตของ op-node (พอร์ต 8655)
ระบบก็แสดงค่าสถานะการซิงก์ทั้งหมดออกมาทันทีเลยครับ

การใช้งานพอร์ต RPC ที่ถูกต้องจึงต้องจำแนกดังนี้:
- **op-geth (RPC Port 8555)**:
  ใช้รันคำสั่งมาตรฐานของกระเป๋าเงินและธุรกรรม (eth_*)
- **op-node (RPC Port 8655)**:
  ใช้รันคำสั่งตรวจสอบสถานะและการทำงานของเชน (optimism_*)

## 1.3 การสื่อสารระหว่างโหนดผ่าน Engine API และ JWT

เมื่อแยกการรันโปรแกรมออกเป็นสองส่วน
หัวใจสำคัญจึงอยู่ที่การสื่อสารข้อมูลระหว่างกัน
โดยใช้ช่องทางของ Engine API และการล็อกความปลอดภัยด้วย JWT ครับ

### Engine API และ JWT Key
ตัว op-node จะส่งบล็อกและคำสั่งปรับปรุงเชนไปยัง op-geth
ผ่านพอร์ต AuthRPC (เช่น พอร์ต 8664)
โดยมีฟังก์ชันหลักในการขับเคลื่อนระบบดังนี้:
- `engine_forkchoiceUpdatedV3`: ส่งข้อมูลเพื่อแจ้งบล็อกล่าสุด
- `engine_newPayloadV3`: ส่งข้อมูลบล็อกใหม่ไปสั่งประมวลผลบน EVM

พอการเข้าถึงเป็นเรื่องสำคัญมากต่อความปลอดภัย
ระบบจึงกำหนดให้มีไฟล์ JSON Web Token (JWT)
ซึ่งก็คือไฟล์ `jwt.txt` ที่บรรจุรหัสผ่านสุ่มขนาด 32 ไบต์
เมื่อตั้งค่าพารามิเตอร์ของทั้งสองโปรแกรมให้ใช้ไฟล์เดียวกันแล้ว
การสื่อสารทั้งหมดจึงจะได้รับการยอมรับและประมวลผลได้ครับ

ตัวอย่างการตั้งค่า JWT บน op-geth:
```bash
op-geth --datadir op-geth-data --networkid 20260619 \
  --http --http.addr 0.0.0.0 --http.port 8555 \
  --http.api eth,net,web3 \
  --authrpc.addr 127.0.0.1 --authrpc.port 8664 \
  --authrpc.jwtsecret jwt.txt \
  --port 30315 --nodiscover --syncmode full &
```

ตัวอย่างการตั้งค่า JWT บน op-node:
```bash
op-node \
  --l2=http://127.0.0.1:8664 \
  --l2.jwt-secret=jwt.txt \
  --l2.enginekind=geth \
  --l1=https://ethereum-sepolia-rpc.publicnode.com \
  --l1.beacon=https://ethereum-sepolia-beacon-api.publicnode.com \
  --rollup.config=rollup.json \
  --rpc.addr=0.0.0.0 --rpc.port=8655 &
```

เมื่อตรวจสอบความเข้ากันได้ของ JWT แล้ว
โหนดทั้งสองฝั่งก็จะเชื่อมต่อกันและรันระบบได้เสถียรครับ

### ไขความลับการซิงก์และระบบ P2P (Orz Correction)
ในระหว่างรันระบบเวิร์กชอป ณ เวลา 08:34 UTC
พี่นัท (P'Nat) ได้ตั้งข้อสังเกตเกี่ยวกับโครงข่าย P2P ว่า:
> "ปัญหาหลักของทั้ง fleet: P2P ต้องเปิด + static peer Nova ถึงจะ sync L2 ได้ is this true?"

ต่อมาในเวลา 08:35 UTC
Orz ได้อธิบายและแบ่งปันข้อเท็จจริงทางสถาปัตยกรรมไว้อย่างชัดเจน
ซึ่งถือเป็นกุญแจสำคัญในการตั้งค่าโหนด L2 เลยครับ

ในมุมมองของตัวประมวลผล (op-geth)
การที่ตั้งค่า flag `--nodiscover` หรือกำหนดขีดจำกัด peer เป็น 0
**ไม่มีผลต่อความสามารถในการซิงก์ข้อมูลบล็อกเลยครับ**
เพราะบล็อกทั้งหมดถูกป้อนเข้าโดยตรงจาก op-node ผ่าน Engine API
ไม่ใช่การวิ่งค้นหาผ่านระบบ devp2p (P2P เดิมของ geth)
เรื่องนี้จึงช่วยคลายความกังวลในการปิดพอร์ต P2P ของเก็ธได้แล้วนะ

แต่เมื่อขยับมาพิจารณาที่ฝั่ง op-node
ระบบการดึงข้อมูลบล็อกหรือ Sync Path จะมีสองทางสำคัญดังนี้:

1. **L1 Derivation (Safe Sync)**:
   ดึงข้อมูลบล็อกดิบที่สะสมอยู่บน L1 (Sepolia) ลงมาสร้างใหม่
2. **P2P Gossip (Unsafe Sync)**:
   ดึงหัวบล็อกล่าสุดจาก Sequencer ของเครือข่ายตรงๆ ผ่าน P2P

เมื่อระบบเพิ่งเริ่มต้นขึ้นในวันแรก
และยังไม่มีกระบวนการรัน op-batcher
เพื่อกวาดธุรกรรม L2 ไปเก็บไว้บนกระดานข่าวของ L1
เส้นทาง L1 Derivation จึงยังไม่มีข้อมูลให้ประมวลผลครับ
พอต้องพึ่งพาการดึงข้อมูลแบบ P2P Gossip เป็นหลัก
หากไปทำการสั่งปิดระบบ P2P (ระบุ flag `--p2p.disable` ใน op-node)
**โหนดลูกก็จะมืดบอดและติดอยู่ที่บล็อก 0 ตลอดกาลเลยนะ**

นี่คือประเด็นสำคัญที่ทีม Weizen และ Vessel
ต้องประสบปัญหาระหว่างเริ่มซิงก์ระบบในตอนแรกครับ
เมื่อแก้ไขโดยการถอด flag ดังกล่าวออก
แล้วเชื่อมต่อเข้าหา Sequencer หลักของ Nova
โดยกำหนด static peer ให้ถูกต้องตามรูปแบบ libp2p multiaddr:

```bash
# ตัวอย่างรูปแบบ libp2p multiaddr สำหรับกำหนด static peer
/ip4/141.11.156.4/tcp/9222/p2p/16Uiu2HAmTZ9fjqstMo...RBeVm
```

พอระบบ P2P เริ่มทำงานประสานกันได้แล้ว
โหนดลูกก็สามารถดึง unsafe head
และอัปเดตบล็อกตามหัวขบวนได้อย่างมีประสิทธิภาพครับ
ต้องจำไว้เสมอด้วยนะว่ารูปแบบที่ใช้เชื่อมต่อ
คือระบบ libp2p multiaddr
ไม่ใช่ enode URL ของ geth ดั้งเดิมครับ
เมื่อนำไปเขียนสลับกัน  ตัวควบคุม P2P ก็จะปฏิเสธคำสั่งทันทีครับ

## บทสรุป

เรื่องการก้าวข้ามจาก Clique PoA มาสู่ OP Stack
ไม่ใช่เรื่องของการเปลี่ยนพารามิเตอร์เพียงบางจุด
แต่เป็นการวางโครงข่ายเชื่อมโยงกันอย่างปลอดภัยเป็นชั้นๆ
เมื่อสามารถประสาน op-node เข้ากับ op-geth
ผ่าน Engine API และคีย์ JWT ได้อย่างเรียบร้อยแล้ว
และตั้งพอร์ต P2P Gossip เชื่อมเข้าหาโหนดหลัก
เครือข่าย L2 ก็จะสามารถเติบโตและผลิตบล็อกต่อไปได้ด้วยดีครับ
