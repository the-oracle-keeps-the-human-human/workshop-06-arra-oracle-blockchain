## 3.1 กลไก block production: 2s block time และ Gas Limit

ในระบบ OP Stack Rollup นั้น 
การผลิตบล็อกไม่ได้อาศัย geth Clique หรือ PoA แบบเดิมนะ

เมื่อเปรียบเทียบกับ L1 แล้ว 
L2 จะสร้างบล็อกด้วยความเร็วที่สม่ำเสมอกว่ามาก
พอถึงรอบเวลาที่กำหนด ตัว sequencer ก็จะสร้างบล็อกขึ้นใหม่ทุก 2 วินาที 
(2s block time)
สำหรับขีดจำกัดแก็สในแต่ละบล็อกนั้น 
ถูกตั้งไว้ที่ Gas Limit 60,000,000 แก็สเลยทีเดียว

เมื่อ Nova รัน sequencer บนเครือข่าย Nova L2 
ก็จำเป็นต้องแก้ไขไฟล์ intent.toml เพื่อลบ customGasToken ออก
และกำหนดค่า rollup.json ให้ตรงกับสเปกที่ตั้งไว้
หากมีธุรกรรมเข้ามาหนาแน่นมากเกินไป 
ก็จะถูกทยอยไปรวมในบล็อกถัดไปนะ

## 3.2 engine_forkchoiceUpdatedV3 และ engine_newPayloadV3

การเชื่อมต่อระหว่าง consensus layer และ execution layer นั้น 
จะใช้โปรโตคอล Engine API

เมื่อ op-node (consensus layer) ทำงานร่วมกับ op-geth (execution layer) 
ก็ต้องมีจุดประสานงานร่วมกัน
พอ op-node ตรวจพบความเคลื่อนไหวจาก L1 origin 
ก็จะเริ่มกระบวนการถอดรหัสธุรกรรม L2
เมื่อต้องการอัปเดตสถานะของบล็อกล่าสุด 
ก็จะเรียกใช้คำสั่ง engine_forkchoiceUpdatedV3 ไปยัง op-geth
จากนั้น เมื่อสร้างบล็อกเสร็จสมบูรณ์ 
จึงส่งข้อมูล payload ไปประมวลผลต่อผ่าน engine_newPayloadV3 นั่นเอง

ในส่วนการรักษาความปลอดภัยนั้น 
ทั้งสองฝั่งก็ต้องตรวจสอบลายเซ็นดิจิทัลผ่าน JWT secret ในไฟล์ jwt.txt
หาก jwt.txt ไม่ถูกต้องหรือไม่ตรงกัน 
op-node ก็จะถูกปฏิเสธสิทธิ์การเชื่อมต่อทันทีครับ

```bash
# ตัวอย่างการสร้าง JWT secret สำหรับการพิสูจน์ตัวตน
openssl rand -hex 32 > jwt.txt
chmod 600 jwt.txt
```

## 3.3 ข้อผิดพลาดและการตรวจวินิจฉัย (RPC `optimism_syncStatus` port collision)

เมื่อนักพัฒนาเริ่มตั้งค่าโหนด 
ก็มักจะเจอกับอุปสรรคเรื่องพอร์ตชนกันบนเซิร์ฟเวอร์ส่วนกลาง

พอ Atom พยายามรันโหนดของตัวเอง 
ก็เกิด port collision ที่พอร์ต P2P 9222 ชนกับ Nova
เมื่อพอร์ตชนกันขึ้นมา ตัวโปรเซสก็ล้มเหลวและ crash ไปสิครับ

แต่นอกจากเรื่องพอร์ตชนแล้ว 
ปัญหาที่พบบ่อยและน่าสับสนที่สุด คืออาการ Diagnostic Blindness
เมื่อ No.10 X หรือ Atom เจอปัญหาว่าข้อมูลโหนดไม่ยอมอัปเดต
ด้วยความสับสน จึงพยายามเรียกหาคำสั่งเช็คสถานะการ sync ผ่านพอร์ต L2 RPC
พอเรียกใช้งาน curl ส่งขอข้อมูล optimism_syncStatus ไปยัง op-geth ที่พอร์ต 8555
ผลลัพธ์ที่ได้กลับมาก็คือการส่งคืนข้อความแสดงความผิดพลาดทันทีนะ

```json
{
  "error": {
    "code": -32601,
    "message": "the method optimism_syncStatus does not exist"
  }
}
```

ทำไมถึงเป็นเช่นนั้นล่ะ?

แต่ความจริงแล้ว วิธีเช็ค sync status นั้น เป็นหน้าที่ของ consensus client
เนื่องจากคำสั่ง optimism_syncStatus ถูกจัดสรรไว้ให้ใช้กับ op-node เท่านั้น
เมื่อเรียกไปที่ op-geth (พอร์ต 8555) ก็จะพัง เพราะไม่มี API ตัวนี้ครับ
พอปรับมาเรียกใช้งานที่ op-node พอร์ต 8655 จึงจะได้รับข้อมูลสถานะที่ถูกต้องนั่นเอง

```bash
# ตัวอย่างการเรียกเช็คสถานะการ sync ที่ถูกต้องไปยัง op-node
curl -X POST http://127.0.0.1:8655 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"optimism_syncStatus","params":[],"id":1}'
```

```bash
# ตัวอย่างการเรียกเช็คความสูงบล็อกปัจจุบันที่ op-geth
curl -X POST http://127.0.0.1:8555 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":1}'
```

การทำงานแบบแยกส่วนแบบนี้ 
หากเข้าใจระบบและจดจำพอร์ตให้ดี 
ก็จะป้องกันความตาบอดในการตรวจวินิจฉัยได้ดีทีเดียวครับ
